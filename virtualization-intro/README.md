# Домашнее задание к занятию «Введение в виртуализацию»

## Задача 1. Запуск «эконом-ВМ» в Yandex Cloud

> Задание предлагает создавать ресурсы через web-интерфейс. Я выполнил то же
> самое через `yc` CLI (результат идентичен), а ниже привёл и инструкцию для
> web-консоли. Все приведённые ниже выводы команд — реальные, с боевой ВМ.

### Параметры созданной «эконом-ВМ»

Конфигурация выбрана по [инструкции по экономии облачных ресурсов](https://yandex.cloud/ru/docs/tutorials/infrastructure-management/budget):
минимум ядер, гарантированная доля vCPU 20 %, минимум RAM, **прерываемая** ВМ
(выключится не позднее чем через 24 часа), сетевой HDD.

| Параметр | Значение |
|----------|----------|
| Имя | `virt-intro-vm` |
| Платформа | `standard-v3` (Intel Ice Lake) |
| vCPU | 2 ядра, **core-fraction 20 %** |
| RAM | 2 ГБ |
| Тип ВМ | **прерываемая** (`preemptible: true`) |
| Загрузочный диск | network-hdd, 30 ГБ (минимум для образа Toolbox) |
| Образ | **Yandex Cloud Toolbox** (Cloud Marketplace), family `toolbox` |
| Публичный IP | `111.88.240.26` (one-to-one NAT, эфемерный) |
| Зона | `ru-central1-a` |

### Шаг 1. Сеть (VPC) и подсеть

```bash
# сеть
yc vpc network create --name virt-intro-net

# подсеть в зоне ru-central1-a
yc vpc subnet create \
  --name virt-intro-subnet-a \
  --network-name virt-intro-net \
  --zone ru-central1-a \
  --range 10.10.0.0/24
```

### Шаг 2. Поиск образа «Yandex Cloud Toolbox» в Marketplace

В web-консоли образ ищется так: «Создать ВМ» → блок **«Выбор образа/загрузочного
диска»** → вкладка **«Cloud Marketplace»** → **«Посмотреть больше»** → поиск
**«Yandex Cloud Toolbox»**.

Через CLI образ находится в публичном каталоге `standard-images`, family `toolbox`:

```bash
$ yc compute image list --folder-id standard-images | grep -i toolbox
| fd83fju1auo6g6nff2g5 | toolbox-v20220812 | toolbox | ... | READY |
```

### Шаг 3. Создание прерываемой ВМ с публичным IP

```bash
yc compute instance create \
  --name virt-intro-vm \
  --zone ru-central1-a \
  --platform standard-v3 \
  --cores 2 --core-fraction 20 --memory 2GB \
  --preemptible \
  --create-boot-disk image-folder-id=standard-images,image-family=toolbox,type=network-hdd,size=30GB \
  --network-interface subnet-name=virt-intro-subnet-a,nat-ip-version=ipv4 \
  --ssh-key ~/.ssh/yc_diplom.pub
```

Результат — ВМ в статусе `RUNNING`, прерываемая, с публичным IP:

```yaml
name: virt-intro-vm
status: RUNNING
platform_id: standard-v3
resources: { memory: 2GB, cores: 2, core_fraction: 20 }
scheduling_policy: { preemptible: true }      # ← прерываемая
network_interfaces:
  - primary_v4_address:
      address: 10.10.0.28
      one_to_one_nat: { address: 111.88.240.26, ip_version: IPV4 }
```

> 📷 `img/01_vm_running.png` — карточка ВМ в web-консоли (статус RUNNING, публичный IP, флаг «Прерываемая»).

### Шаг 6. Прерываемость и оценка расходов

ВМ создана как **прерываемая** (`--preemptible`): Yandex Cloud вправе остановить
её в любой момент, но **гарантированно выключит не позднее чем через 24 часа**.
Это и есть основной приём экономии — прерываемые ВМ значительно дешевле обычных.

Согласно заданию ВМ оставлена работать до самостоятельного выключения; ещё через
сутки нужно открыть [биллинг](https://console.yandex.cloud/billing) →
выбрать платёжный аккаунт → вкладка **«Детализация»** (фильтр **«По продуктам»**)
и оценить график потребления.

> 📷 `img/03_billing.png` — график детализации расходов по продуктам (заполнить через ~1–2 суток).

После выполнения задания ресурсы удаляются:

```bash
yc compute instance delete --name virt-intro-vm
yc vpc subnet delete --name virt-intro-subnet-a
yc vpc network delete --name virt-intro-net
```

> последующих домашних работах курса.

## Задача 2. Выбор типа платформы под задачу

Однозначно «правильного» ответа нет — выбор зависит от бюджета, компетенций
команды, требований ИБ/законодательства и надёжности. Ниже — обоснованный выбор
под каждый сценарий.

| Задача | Рекомендуемая платформа | Почему |
|--------|-------------------------|--------|
| **Высоконагруженная БД MySQL, критичная к отказу** | **Физические серверы** (bare-metal) | БД критична к latency дисков и предсказуемости I/O. На «голом железе» нет накладных расходов гипервизора и «эффекта шумного соседа», доступны NVMe напрямую. Отказоустойчивость закрывается на уровне самой БД (репликация master-slave/Galera) + резервированием железа. |
| **Различные web-приложения** | **Виртуализация уровня ОС (контейнеры)** | Web-приложения легковесны, их много, они часто масштабируются и обновляются. Контейнеры дают минимальные накладные расходы, высокую плотность размещения, быстрый старт и удобный CI/CD. Изоляция в пределах одного ядра здесь достаточна. |
| **Windows-системы для бухгалтерии** | **Паравиртуализация / полная аппаратная виртуализация** (KVM, Hyper-V, VMware) | Контейнеры Linux не запустят Windows-приложения 1С/бухгалтерии. Нужна полноценная гостевая ОС Windows с собственным ядром. Гипервизор даёт нужную изоляцию, снапшоты, простое резервное копирование рабочих мест. |
| **Высокопроизводительные расчёты на GPU** | **Физические серверы** (или ВМ с GPU-passthrough) | Для максимума производительности GPU пробрасывается напрямую (PCI passthrough) либо используется bare-metal — это минимизирует потери на виртуализации и даёт полный доступ к CUDA/драйверам. Прослойка гипервизора здесь — лишние накладные расходы. |

**Критерии выбора в общем виде:**
- *Производительность и предсказуемость I/O* → ближе к «железу» (bare-metal, passthrough).
- *Плотность размещения и скорость деплоя* → контейнеры (виртуализация уровня ОС).
- *Необходимость чужой ОС / сильной изоляции* → полная/паравиртуализация.
- *Стоимость и утилизация* → виртуализация повышает утилизацию железа, но добавляет overhead.

## Задача 3. Выбор системы управления виртуализацией

| Сценарий | Решение | Обоснование |
|----------|---------|-------------|
| **1. ~100 ВМ Linux+Windows, общие задачи, преимущественно Windows; нужны программные балансировщики, репликация и автоматический бэкап** | **VMware vSphere + vCenter** (как вариант — Microsoft Hy-V + System Center / Failover Clustering) | Зрелая корпоративная платформа: vMotion, HA, DRS, репликация (vSphere Replication), встроенные/интегрированные бэкап-решения (Veeam), балансировка нагрузки (NSX / NLB). Отлично работает и с Windows, и с Linux, хорошо масштабируется на сотни ВМ. Для чисто Windows-инфраструктуры альтернатива Hyper-V даёт лучшую интеграцию с AD и лицензионную выгоду. |
| **2. Самое производительное бесплатное open source для 20–30 серверов Linux+Windows** | **Proxmox VE** (KVM + LXC) | Бесплатный open source, веб-интерфейс «из коробки», кластеризация, HA, живая миграция, ZFS, встроенный бэкап (PBS). KVM показывает производительность, близкую к железу, и тянет как Linux, так и Windows-гости. Идеален для небольших инфраструктур без бюджета на лицензии. |
| **3. Бесплатное, максимально совместимое и производительное под Windows-инфраструктуру** | **Microsoft Hyper-V (Hyper-V Server / роль Windows Server)** | Нативный гипервизор Microsoft: максимальная совместимость с Windows-гостями, интеграционные службы, поддержка драйверов, бесшовная работа с Active Directory и System Center. Тип-1 гипервизор → высокая производительность. Сам Hyper-V бесплатен. |
| **4. Окружение для тестирования ПО на нескольких дистрибутивах Linux** | **VirtualBox** (или Vagrant поверх него) | Бесплатный, кроссплатформенный, удобен для разработчика на рабочей станции: снапшоты, быстрое создание/удаление ВМ, импорт готовых образов. В связке с Vagrant — мгновенное разворачивание матрицы дистрибутивов из одного `Vagrantfile`. Для эпизодического тестирования не нужна тяжёлая серверная платформа. |

## Задача 4. Гетерогенная среда виртуализации

**Гетерогенная среда** — одновременное использование нескольких систем
управления виртуализацией (например, VMware + Proxmox + Hyper-V).

### Проблемы и недостатки

1. **Рост сложности эксплуатации.** Разные интерфейсы, API, концепции и форматы
   дисков (VMDK / QCOW2 / VHDX). Администраторам нужно знать каждую платформу.
2. **Несовместимость форматов и затруднённая миграция.** Перенос ВМ между
   платформами требует конвертации образов и часто простоя.
3. **Зоопарк инструментов.** Резервное копирование, мониторинг, автоматизация
   (IaC) и оркестрация приходится настраивать отдельно под каждую систему — нет
   «единого окна».
4. **Рост требований к компетенциям и штату.** Нужны специалисты по каждой
   платформе → выше затраты на персонал и обучение, риск «bus factor».
5. **Усложнение ИБ и комплаенса.** Каждая платформа — отдельная поверхность атаки,
   свои политики, патч-менеджмент и аудит. Сложнее поддерживать единые стандарты
   безопасности.
6. **Сложное лицензирование и учёт затрат.** Разные модели лицензий и поддержки,
   труднее прогнозировать TCO.
7. **Размытая ответственность и поддержка.** При инциденте сложнее определить
   виновную прослойку; вендоры могут «переводить стрелки».

### Как минимизировать риски

- **Единый слой управления/мониторинга** поверх платформ (например, CloudStack,
  OpenStack, Terraform с провайдерами под каждый гипервизор, Zabbix/Prometheus
  для общего мониторинга).
- **Стандартизация через IaC** — описывать инфраструктуру декларативно (Terraform,
  Ansible), чтобы скрыть различия платформ за единым кодом.
- **Единая стратегия резервного копирования** (например, Veeam, поддерживающий
  несколько гипервизоров).
- **Документация и регламенты** — чёткие runbook'и, схемы, разграничение зон
  ответственности.
- **Обучение и кросс-компетенции** команды, чтобы снизить bus factor.
- **Чёткое обоснование зон применения** каждой платформы (где и почему она
  используется), а не стихийное «исторически сложилось».

### Стал бы я создавать гетерогенную среду?

**По возможности — нет.** Если позволяют требования, я выбрал бы **одну**
платформу: это проще в эксплуатации, дешевле в поддержке, единые инструменты,
бэкап и мониторинг, ниже требования к штату.

Гетерогенная среда **оправдана только тогда**, когда это вынужденная мера:
- разные задачи объективно требуют разных платформ (например, Hyper-V под
  Windows-парк + KVM/Proxmox под Linux-нагрузки);
- нужно избежать vendor lock-in или снизить лицензионные затраты;
- среда сложилась исторически в результате слияний/наследия (legacy).

В таком случае гетерогенность нужно **сознательно проектировать и закрывать
единым слоем управления, мониторинга и автоматизации**, а не оставлять как
стихийный «зоопарк».
